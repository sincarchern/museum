1. Scrum 的核心角色

Scrum 有三個角色，確保團隊協作高效：

  • 產品負責人（Product Owner）：決定產品待辦清單（Product Backlog）的優先順序，確保團隊創造最大價值。
  • Scrum 團隊（Scrum Team）：負責實際交付產品增量（Increment）的跨功能團隊，通常包括開發人員。
  • Scrum 大師（Scrum Master）：協助團隊實踐 Scrum，移除障礙並促進流程改進。

2. Scrum 的專案項目

Scrum 框架中有三個主要的專案項目：

  • 產品待辦清單（Product Backlog）：產品的所有需求清單，由產品負責人持續更新與優化。
  • 短期迭代待辦清單（Sprint Backlog）：每次短期迭代（Sprint）期間要完成的工作，由團隊選取自產品待辦清單。
  • 增量（Increment）：每個短期迭代產生的可交付價值，應滿足「完成的定義」（Definition of Done）。

3. Scrum 的活動

Scrum 中有四個主要活動：

  • 短期迭代計劃會議（Sprint Planning）：確定短期迭代目標和將完成的工作。
  • 每日站會（Daily Scrum）：短暫的每日會議，幫助團隊同步進度並解決問題。
  • 短期迭代檢視（Sprint Review）：在短期迭代結束時展示增量並獲取反饋。
  • 短期迭代回顧（Sprint Retrospective）：反思團隊合作與流程，提出改進計劃。

4. Scrum 的價值觀

Scrum 的五大核心價值觀：

  • 專注（Focus）
  • 開放（Openness）
  • 承諾（Commitment）
  • 尊重（Respect）
  • 勇氣（Courage）

實踐建議與範例

1. 實踐建議

  • 建立「完成的定義」：明確列出什麼樣的工作才算完成，確保團隊對交付標準達成共識。
  • 規劃合理的短期迭代長度：通常為 1～4 週，選擇適合團隊的週期。
  • 每日站會時間控制在 15 分鐘內：每個人回答三個問題，避免進行技術討論。
  • 透明化工作進度：使用工具（如 Jira、Trello、看板牆）可視化進度。
  • 短期迭代回顧要聚焦改進：鼓勵團隊成員分享並設定具體的改進行動。
  • 避免產品負責人過度干預短期迭代進行中：在短期迭代期間避免頻繁更改需求。

2. 實踐範例

範例 1：軟體開發團隊的 Sprint

短期迭代目標：在 2 週內完成用戶註冊頁面功能並整合至現有系統。

  • 設計用戶界面原型
  • 撰寫後端 API
  • 實現前端功能並與 API 整合
  • 撰寫自動化測試
  • 確保所有功能通過測試

範例 2：博物館活動專案管理

短期迭代目標：在 3 週內完成博物館「特展導覽活動」的策劃與報名系統搭建。

  • 撰寫活動企劃書
  • 制作活動宣傳海報
  • 與外部導覽講師確認檔期
  • 搭建線上報名系統
  • 測試並確認報名流程順暢

實踐細節

1. 短期迭代規劃（Sprint Planning）

時間長度：短期迭代開始時舉行，通常控制在 2 小時內。

2. 每日站會（Daily Scrum）

每位成員回答三個核心問題，並分享進度與障礙。

3. 短期迭代檢視（Sprint Review）

展示完成的產品增量，接受利益相關者的反饋。

4. 短期迭代回顧（Sprint Retrospective）

討論哪些實踐該開始、停止或繼續。

工具建議

  • Jira：完整的 Scrum 工具，支援產品待辦清單管理、短期迭代規劃、每日站會和報表分析。
  • Trello：簡單易用的看板管理工具。
  • ClickUp：結合任務清單、甘特圖和看板。
  • Slack：團隊即時溝通工具。
  • Miro：白板工具，適合遠程協作。
